业务方可以将数据输出到日志中，系统对接日志来异步采集数据。这使得数据采集不会影响业务处理主流程。 这种方式风险在于： （1）业务风险防范 通过在页面上埋入脚本来监测用户的操作，特别是点击操作和键盘操作，将检测到的用户行为异步发送到服务器端。这可以侦测到用户在页面停留时间，鼠标点击的区域等信息，由此可以推断用户偏好，情绪等信息。可以分析用户的行为偏好，对用户的身份

（2）系统风险： 需要规范日志的格式，否则每个系统一套日志格式，会导致对接工作量巨大。 保持日志的稳定性。一旦代码被修改，打印日志的代码被删除了，会导致日志数据无法采集的风险。 需要注意日志采集系统的可靠性。目前主流的采集框架都有可能会丢失日志。虽然从我们使用的情况来还未发生这种事情，但不排除这个风险。

从技术上来说，日志采集的框架主要框架有:ELK（Elastic + Logstash + Kibana）， Logstash 驻留在日志输出端采集日志，并发送到Elastic 服务器上。 Kibana则是一个日志分析的工具；
